# Learning Log

- Diagnose loop: the Settings recorder uses the `KeyboardShortcuts` local event monitor, while Foreman also has its own Carbon `RegisterEventHotKey` handler for the active toggle chord. `KeyboardShortcuts.isPaused` only pauses that package's handlers, not Foreman's custom Carbon handler.
- Hypotheses considered:
  1. Foreman's custom Carbon hotkey remains registered while the recorder is active, so the current toggle chord can be consumed by the global handler instead of the recorder.
  2. Re-rendering `SettingsView` after shortcut changes replaces the recorder mid-edit.
  3. The floating `NSPanel` prevents the `RecorderCocoa` field from becoming first responder.
- First fix target was (1): observe the recorder-active notification and temporarily unregister Foreman's Carbon handler while recording. It improved the recorder handoff but still allowed a transient `eventHotKeyExistsErr` (`OSStatus -9878`) after recording Cmd+Shift+F.
- Revised fix: suspend Foreman's Carbon hotkey for the whole Settings panel lifetime and re-register only when Settings closes. That avoids racing the recorder's save/blur/end-recording sequence and makes the Settings status explicit while editing. The user correctly pointed out that Settings must be open to change the shortcut; the intended behavior is only that the global toggle is paused while Settings is open, not that recording waits until after close.
- A second real issue appeared as a control diagnostic: one malformed `tmux list-panes` line (`missing session_name`) could collapse the whole macOS app inventory to zero agents. Foreman now skips malformed pane records instead of failing the entire inventory load.
- Carbon can still report transient `eventHotKeyExistsErr` (`OSStatus -9878`) when re-registering immediately after Settings closes, so the hotkey controller briefly retries that specific conflict for environment override mode.
- External research confirmed the common SwiftUI pattern is to use Sindre Sorhus's `KeyboardShortcuts.Recorder` together with `KeyboardShortcuts.onKeyUp`/`onKeyDown`, not a separate app-owned Carbon registration for the same persisted shortcut. Foreman's normal persisted shortcut path now uses `KeyboardShortcuts` for both recording and global handling; the custom Carbon path is retained only for `FOREMAN_OVERLAY_HOTKEY` development overrides.
- `KeyboardShortcuts.reset(.name)` means restore the default shortcut, not clear it. Settings now exposes separate `Clear Shortcut` and `Restore Default` actions with explanatory copy.
- Clearing initially failed in the gauntlet because `configureHotkey()` treated `getShortcut == nil` as first launch and restored the default. It now checks whether the UserDefaults key exists, so a disabled/cleared shortcut stays cleared.
- Default hotkey was first changed from Ctrl+F to Cmd+Shift+F, but `CopySymbolicHotKeys` shows an enabled system symbolic hotkey for key code F with modifiers 768, and `KeyboardShortcuts.Recorder` correctly warns that Cmd+Shift+F is system-wide. The default is now Cmd+Option+F instead.
- Settings snapshots can pick up the operator's persisted KeyboardShortcuts value. The snapshot renderer now temporarily resets the shortcut to the default while rendering settings, then restores the prior local preference.
- macOS GUI launches may omit locale variables. Without `LC_CTYPE`/`LANG`, tmux 3.6a replaced actual tab separators in `list-panes -F` output with underscores, making Foreman's parser treat every pane line as malformed. `ProcessRunner` now seeds UTF-8 locale variables for GUI-launched subprocesses.
- Reviewer caught that skipping every tmux parse error would mask real format regressions. The parser now skips only non-tabbed noise lines, errors if all records are malformed, and still fails truncated tabbed records.
- Reviewer also caught that `KeyboardShortcuts.isEnabled` is not OS-level registration proof. Settings now says the recorder shortcut is configured, not Carbon-registered; normal persisted shortcut handling delegates to `KeyboardShortcuts` instead of claiming separate Carbon status.
- Rolled `HK_NATIVE_MIGRATION_PLAN.md` into this branch: new work should use HK lifecycle and compact `.ai/hk/<work-id>/` exports. `.ai/plans/**` remains historical/legacy evidence.
