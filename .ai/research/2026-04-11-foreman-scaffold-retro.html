Foreman did not outgrow the scaffold by accident.

The scaffold gave a good outer loop. Foreman started moving fast only after the repo added a product-specific spec, a real architecture record, a chunked plan, and heavy validation lanes for tmux, compiled-binary flows, UX evidence, and real harness drills.

What Worked

  • Stable task contract from day 0: plan, check, verify, ci.
  • Spec-first vertical slicing produced a clean build order.
  • Validation culture moved from fake seams to real boundaries instead of stopping early.
  • Recurring lessons were eventually promoted into durable docs.

Where It Lacked

  • Generated docs were placeholder-heavy and not shaped for a harness TUI.
  • The architecture template was biased toward service backends, not reducer/render/tmux seams.
  • Plan templates were too thin for a multi-stage rebuild.
  • The Rust starter was a hello-world binary plus Greeter.
  • The generic verify lane was too weak for harness engineering.

What Needed Real Discovery

  • Codex hook behavior changed between non-interactive mode and tmux TTY.
  • Detached tmux alternate-screen capture was flaky.
  • CI exposed shell portability, window-index assumptions, and startup races.
  • UX trust gaps only became obvious under operator-style walkthroughs.

Best Scaffold Upgrade

Add an explicit tui-harness archetype. That would have removed the largest avoidable churn without pretending empirical harness work can be templated away.

What Would Have Made It More One-Shot

Change from the start Why it matters
Product-specific AGENTS, SPEC, and architecture Would have removed early placeholder rewrite churn and made boundaries explicit on day 0.
Complex-plan mode with traceability and validation-layer prompts The first real Foreman plan had to invent that structure anyway.
Named heavy validation lanes verify-release, verify-ux, and verify-native became core, not optional polish.
Harness fixture helpers Retry, alternate-screen fallback, and atomic signal writes were required to make the real-boundary story stable.
Workflow docs in addition to architecture Foreman needed a durable place for rough edges, validation rules, and lesson promotion.

Recommended `agent-scaffold` Upgrades

  • Add a tui-harness archetype, not just stack + shape.
  • Generate docs/tour.md and docs/workflows.md.
  • Add a complex-plan template with traceability, exit criteria, and a one-shot retrospective prompt.
  • Let archetypes register named heavy lanes and artifact/report conventions.
  • Scaffold tmux and harness fixture helpers for the archetype.
  • Provide optional companion-binary stubs and release packaging hooks.
  • Harden Docker validation defaults when a Dockerfile is scaffolded.
Bottom line: the scaffold gave Foreman a solid shell. The missing piece was not "more generic intelligence." It was a repo archetype that already assumed reducer-driven TUI work, external harness boundaries, and evidence-heavy validation.
